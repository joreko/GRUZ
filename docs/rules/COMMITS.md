# ГРУЗ — Правила коммитов

## Формат

```
XXXX
тип(область): краткое описание на русском
```

Заголовок коммита — счётчик. Тело — Conventional Commits на русском.
В командной строке:
```
git commit -m "XXXX" -m "тип(область): описание"
```

### Заголовок — счётчик
```
XXXX
```
Просто число, без текста (текущий стиль проекта — последний коммит на beta).

### Тело — Conventional Commits на русском
```
тип(область): что сделано — файл.rs / Component.svelte
```
Каждая строка — конкретное изменение: что именно, где и в каком файле.
Не писать общие фразы вроде «рефакторинг кода» или «исправления багов» —
указывать конкретный модуль, функцию, поведение и файл через ` — `.

**Формат указания файла:**
- Rust: `— mod.rs`, `— process.rs`, `— tasks.rs`
- Svelte: `— DownloadPage.svelte`, `— TitleBar.svelte`
- TypeScript: `— commands.ts`, `— changelog.ts`
- Конфиги: `— tauri.conf.json`, `— Cargo.toml`

**Типы (пользовательские — видны в changelog):**
| Тип | Когда использовать |
|---|---|
| `добавлено` | новый функционал |
| `исправлено` | исправление бага |
| `улучшено` | UI/визуальные изменения |
| `быстрее` | оптимизация производительности |
| `удалено` | удаление функционала |

**Лирическое описание релиза (обязательно для тегового коммита):**
```
описание: одна фраза о сути релиза — что это за обновление по духу
```
Пишется в тело тегового коммита. Отображается в карточке релиза в приложении.
Пример: `описание: переработан загрузчик — теперь он умнее и не теряет файлы`

**Типы (технические — скрыты от пользователя):**
| Тип | Когда использовать |
|---|---|
| `рефакт` | рефакторинг без изменения поведения |
| `доки` | изменения документации |
| `сборка` | зависимости, конфиги, служебное |
| `тесты` | тесты |

**Области (примеры под стек ГРУЗ):**

Область отображается пользователю как бейджик в changelog — выбирай то,
что пользователь поймёт без знания кода.

**UI-страницы (привязаны к конкретному экрану):**
| Scope | Когда использовать | Бейджик в UI |
|---|---|---|
| `download` | страница загрузки: URL, форматы, параметры, кнопка скачать | Загрузка |
| `queue` | страница очереди задач | Очередь |
| `gallery` | галерея скачанных файлов | Галерея |
| `history` | история загрузок | История |
| `settings` | страница настроек | Настройки |
| `save-settings` | страница шаблонов сохранения | Сохранение |
| `updates` | страница обновлений | *(скрыт — очевидный контекст)* |
| `sidebar` | боковая навигация | Навигация |
| `titlebar` | заголовок окна, мысли оркестратора | Шапка |

**Функциональные модули (движок, не UI):**
| Scope | Когда использовать | Бейджик в UI |
|---|---|---|
| `installer` | установщик / инсталлятор | Установщик |
| `downloader` | движок скачивания (yt-dlp, ffmpeg, прогресс) | Загрузчик |
| `orchestrator` | координатор задач, воркеры, логика очереди | Оркестратор |
| `queue` | in-memory очередь задач (Rust) | Очередь |
| `ytdlp` | управление бинарником yt-dlp | yt-dlp |
| `db` | база данных, миграции, схема | База данных |
| `channel-prefs` | настройки каналов | Каналы |
| `session` | сессия пользователя | Сессия |

**Технические (архитектура, инфраструктура):**
| Scope | Когда использовать | Бейджик в UI |
|---|---|---|
| `bridge` | IPC команды/события Tauri | *техническое* |
| `commands` | Tauri-команды (Rust) | *техническое* |
| `stores` | Svelte-сторы | *техническое* |
| `design` | дизайн-система, токены, CSS | Дизайн |
| `ui` | общий интерфейс без привязки к странице | Интерфейс |
| `tauri` | конфигурация Tauri, capabilities, CSP | *техническое* |
| `ci` | GitHub Actions, сборка, релиз | *техническое* |
| `deps` | зависимости Cargo.toml / package.json | *техническое* |

**Правило:** если изменение видно пользователю на конкретной странице —
используй scope этой страницы. Технические scope — только если изменение
не привязано к конкретному экрану.

### Примеры
```
41
добавлено(queue): ручная перестановка задач drag-n-drop — QueuePage.svelte
```
```
42
исправлено(downloader): корректно парсить прогресс при слиянии аудио+видео — process.rs
```
```
43
рефакт(orchestrator): вынести scheduler в отдельный модуль без смены поведения — mod.rs
```

## Версионирование

```
МА.МИ.СЧЁТЧИК
41 (пример — номер последнего коммита на момент написания)
│  │  └── счётчик: +1 за каждое смысловое изменение в коммите
│  └───── минор: новый функционал
└──────── мажор: кардинальное изменение / публичный релиз
```

### Что считается одним изменением (+1)
Каждая строка тела = одно смысловое изменение:
- каждая новая функция/метод (даже приватная вспомогательная)
- исправление одного бага
- рефакторинг одного конкретного модуля/функции
- новый файл/компонент/команда
- каждая новая зависимость (Cargo.toml / package.json)
- каждый новый документ в docs/

**Каждая функция — отдельная строка.** «добавлены методы A, B, C» = ошибка,
нужно три строки.

**Не считается отдельным изменением:**
- форматирование внутри уже изменённого файла
- переименование переменной как часть более крупного изменения

Пример: предыдущая версия `40`, в новом коммите 3 строки тела → `43`.

## Changelog и счётчик (+N)

Приложение показывает пользователю сколько изменений вышло с момента установки:
`v0.0.1 (+36)` — где `36` это сумма пользовательских строк тела всех коммитов
вышедших после установки.

**Как считается (+N):**
- При установке запоминается текущий счётчик (заголовок последнего коммита)
- Берутся все коммиты вышедшие после этого числа
- Фильтруются только пользовательские типы: `добавлено`, `исправлено`, `улучшено`, `быстрее`
- Суммируется количество строк тела этих коммитов

**Список изменений в приложении:**
- Сгруппирован по semver-версии (`v0.0.1`, `v0.0.2`, ...)
- Показывает только строки пользовательских типов
- Технические коммиты (`рефакт`, `сборка`, `тесты`, `доки`) скрыты полностью

## Ветки

- `beta` — вся разработка. Все обычные коммиты идут сюда.
- `main` — только релизные коммиты с тегом. Прямые коммиты на main запрещены.

Поток: коммит на `beta` → при релизе cherry-pick в `main` → тег → CI.

## Виды коммитов

### Обычный коммит
Фиксирует изменения в коде. Коммитится на `beta`.

### Релизный коммит
Создаётся когда набралось достаточно изменений для выпуска. Порядок:
1. Сделать коммит на `beta` как обычно
2. Cherry-pick в `main`: `git cherry-pick <sha>`
3. Создать тег: `git tag v0.0.129`
4. Запушить: `git push origin main && git push origin v0.0.129`

CI автоматически вписывает версию из тега в бинарник и публикует Release на GitHub.
Формат тега: `v0.0.X` — patch это счётчик последнего коммита.

```
git cherry-pick <sha>
git tag v0.0.X
git push origin main
git push origin v0.0.X
```

## Правила

- Коммитить **только с явного разрешения пользователя** — не проявлять инициативу,
  не предлагать коммит самостоятельно, не коммитить без команды.
- Перед коммитом показать пользователю оформление (версию + все строки тела)
  и дождаться подтверждения.
- Пуш — **только с явного разрешения пользователя**. Агент не пушит самостоятельно.
- Коммит = смысловой блок: завершённый модуль, фича, рефакторинг или набор
  документов. Не каждый файл, не каждое изменение.
- Один коммит = одна логическая единица. Не мешать несвязанные изменения.
- Не коммитить секреты, бинарники моделей, lock-файлы данных.
- Стейджить конкретные файлы, не `git add .` вслепую.
